Sustentación Pregunta 3 (6 puntos)
User Profiles - Diagrama de Clases del Dominio
Criterios y Sustento de decisión

He seleccionado el bounded context User Profiles dentro de la arquitectura propuesta para EduFlow Digital Corp. La elección de este BC responde a que, si bien no es el núcleo transaccional más complejo de la plataforma (como podría ser Subscription & Billing Management), representa de manera clara y didáctica los conceptos fundamentales de Domain-Driven Design solicitados en la pregunta: un Aggregate Root bien definido, Value Objects inmutables, y reglas de negocio encapsuladas. Adicionalmente, al ser un BC de soporte (supporting domain) con menor riesgo de error en su modelado, permite enfocarse en la correcta aplicación de los principios DDD sin la complejidad añadida de otros BCs como Billing o Workflow, lo cual es estratégico en un contexto de examen con tiempo limitado. El caso de estudio no exige seleccionar un BC core, por lo que esta elección es completamente válida y alineada con el enunciado.

En el diagrama de clases del dominio, he considerado UserProfile como el Aggregate Root, que representa la información principal de un usuario dentro de la plataforma. Esta clase tiene una relación de composición de 1 a 1 con Email, implementado como un Value Object por ser inmutable y carecer de identidad propia fuera del Aggregate Root. La clase Email encapsula la lógica de validación de formato a través de su método isValid(), aplicando así el principio de encapsulamiento de DDD. Además, UserProfile se relaciona con la enumeración Language, que representa los idiomas soportados por la plataforma (ENGLISH, SPANISH, FRENCH), la cual se modela como un enum por tratarse de valores fijos y conocidos en tiempo de diseño. El Aggregate Root expone métodos de negocio representativos como createProfile(), changeEmail(), activate() y deactivate(), los cuales reflejan operaciones del mundo real dentro del dominio de gestión de perfiles. Se ha aplicado la convención de nomenclatura en inglés con Upper-Camel-Case para clases y Lower-Camel-Case para atributos y métodos, cumpliendo con los estándares UML y de lenguajes orientados a objetos como Java o C#. Esta estructura garantiza que todas las operaciones sobre el perfil de usuario pasen a través del Aggregate Root, manteniendo la consistencia transaccional y la invariante de negocio de que un usuario inactivo no puede realizar cambios en su perfil.

Sustentación Pregunta 4 (6 puntos)
User Profiles - CQRS (ChangeEmailCommand)
Criterios y Sustento de decisión

Para la aplicación de CQRS dentro del mismo bounded context User Profiles, he seleccionado el comando ChangeEmailCommand por ser significativo desde el punto de vista del negocio y por ilustrar perfectamente la separación entre operaciones de escritura (command) y consultas (query). Este comando involucra reglas de negocio no triviales: el nuevo email debe tener un formato válido, no puede ser igual al actual, y el usuario debe estar activo para poder cambiar su correo electrónico. Adicionalmente, el cambio de email es una operación que potencialmente debe notificar a otros BCs como Billing o Notification Services, lo que permite demostrar la integración con Event-Driven Architecture más adelante. Se ha elegido este comando en lugar de otros más simples (como UpdateProfileName) precisamente para mostrar la riqueza de reglas de negocio que incluso un BC de soporte puede contener, demostrando que la elección de User Profiles no sacrifica la capacidad de aplicar patrones avanzados como CQRS.

El diagrama presenta ChangeEmailCommand como un comando inmutable, con un constructor que recibe los parámetros userId y newEmail, y únicamente expone métodos getter para acceder a estos valores. Su correspondiente ChangeEmailCommandHandler implementa la interfaz genérica CommandHandler<ChangeEmailCommand> y recibe por inyección de dependencias un UserRepository. El método handle() del handler ejecuta la siguiente secuencia: (1) recupera el Aggregate Root UserProfile usando el repositorio, (2) ejecuta el método de dominio changeEmail() que contiene las reglas de negocio (validación de usuario activo y email diferente), y (3) persiste el cambio mediante save(). Esta separación de responsabilidades es fundamental en CQRS: el comando solo transporta datos, el handler orquesta la ejecución, y el Aggregate Root aplica las reglas de negocio. El repositorio se modela como una interfaz para mantener el desacoplamiento y permitir diferentes implementaciones (base de datos, caché, etc.). Se ha respetado la nomenclatura en inglés con sufijos Command y Handler para identificar claramente el patrón, y se ha utilizado Lower-Camel-Case para métodos como handle() y getUserId(). Finalmente, aunque el diagrama simplificado no incluye eventos de dominio por razones de tiempo, en una implementación completa el handler publicaría un EmailChangeRequestedEvent para que otros BCs reaccionen, alineándose así con el requisito de Event-Driven Architecture mencionado en el caso.
